52  // SDAG code follows. See http://charm.cs.illinois.edu/manuals/html/
53  // charm++/manual.html, Sec. "Structured Control Flow: Structured Dagger".
54 
55  // High-level overview of the dependency and asynchronous call structure
56  // ---------------------------------------------------------------------
57  //
58  // Directed Acyclic Graph (DAG):
59  // -----------------------------
60  //
61  // ChSol -- HypreSol -- FillSol -- AsmSol -
62  // \
63  // ChLhs -- HypreLhs -- FillLhs -- AsmLhs -- Sol - Upd
64  // /
65  // ChRhs -- HypreRhs -- FillRhs -- AsmRhs -
66  //
67  // DAG legend:
68  // ChSol - Chares contribute their solution vector nonzeros
69  // ChLhs - Chares contribute their left hand side matrix nonzeros
70  // ChRhs - Chares contribute their right hand side vector nonzeros
71  // HypreSol - Convert solution vector to Hypre format
72  // HypreLhs - Convert left hand side matrix to Hypre format
73  // HypreSol - Convert right hand side vector to Hypre format
74  // FillSol - Fill/set solution vector
75  // FillLhs - Fill/set left hand side matrix
76  // FillRhs - Fill/set right hand side vector
77  // AsmSol - Assemble solution vector
78  // AsmLhs - Assemble left hand side matrix
79  // AsmRhs - Assemble right hand side vector
80  // Sol - Solve linear system
81  // Upd - Update solution vector (propagate new solution back to workers)
82  //
83  // Interpretation of the above DAG
84  // -------------------------------
85  // Control flow is from left to right and top to bottom.
86  //
87  // We start out with all Performer (worker) chares contributing their
88  // portion of the solution (unknown) vector, the nonzero values of the
89  // left hand side matrix, and the right hand side vector to those Linear
90  // System Merger group branch (one per CPU) they happened to reside on.
91  // These steps are denoted by ChSol, ChLhs, and ChRhs in the graph above,
92  // performed by the member functions charesol(), charelhs(), and
93  // charerhs(), respectively.
94  //
95  // Once each of the tasks, ChSol, ChLhs, ChRhs, (not all at the same time
96  // but separately) are done, we continue by converting these data
97  // structures to a format that Hypre expects, done in hypresol(),
98  // hyprelhs(), and hyprerhs(), denoted by similars labels in the graph.
99  // These functions basically flatten the C++ data structures to C-style
100  // arrays and linked arrays (for the matrix).
101  //
102  // Once the tasks, HypreSol, HypreLhs, HypreRhs, (not all at the same time
103  // but separately) are done, we continue by assigning the flattened data
104  // structures (more precisely, their pointers) to the Hypre vectors and
105  // matrix. These are denoted FillSol, FillLhs, and FillRhs in the DAG and
106  // correspond to the functions sol(), lhs(), and rhs().
107  //
108  // Once the tasks, FillSol, FillLhs, FIllRhs, (not all at the same time,
109  // but separately) are, we continue by calling the Hypre "assembly"
110  // routines for the solution vector, the left hand side matrix, and the
111  // right hand side vector. These are denoted by AsmSol, AsmLhs, and AsmRhs
112  // in the DAG, and performed by the member functions assemblesol(),
113  // assemblelhs(), and assemblerhs(), respectively.
114  //
115  // Once the assembly of the solution (unknown) vector, the left hand side
116  // matrix, and the right hand side vector are all done (on a CPU), we call
117  // the solve() member function which solves the linear system calling
118  // Hypre and also updates the solution, i.e., propagates the new solution
119  // back to the worker (Performer) chares.
120  //
121  // Note that Sol and Upd are both partial synchronization points, since
122  // they can happen at different times on different CPUs.
123
