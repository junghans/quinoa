63  // SDAG code follows. See http://charm.cs.illinois.edu/manuals/html/
64  // charm++/manual.html, Sec. "Structured Control Flow: Structured Dagger".
65 
66  // High-level overview of the dependency and asynchronous call structure
67  // ---------------------------------------------------------------------
68  //
69  // Directed Acyclic Graph (DAG):
70  // -----------------------------
71  //
72  // -- ChSol -- HypreSol -- FillSol -- AsmSol --
73  // / / \
74  // ChRow --- Ver --- ChLhs -- HypreLhs -- FillLhs -- AsmLhs -- Sol - Upd
75  // /| \ / | / /
76  // / | \ | |--- /
77  // / | \ | | /
78  // ChBCs -- | - ChRhs -- HypreRhs -- FillRhs -- AsmRhs -
79  // | |/ / /
80  // |-- HypreRow ------------------
81  // | |
82  // |-- VerRow |
83  // \ /
84  // -- VerBCs --
85  //
86  // DAG legend:
87  // ChRow - Chares contribute their global row (mesh node) IDs
88  // ChBCs - Chares offer their global node IDs at which they can set BCs
89  // ChSol - Chares contribute their solution vector nonzeros
90  // ChLhs - Chares contribute their left hand side matrix nonzeros
91  // ChRhs - Chares contribute their right hand side vector nonzeros
92  // Ver - Start optional verifications and HypreRow (see below)
93  // VerRow - Optional verification ensuring consistent row IDs
94  // VerBCs - Optional verification ensuring consistent BC node IDs
95  // HypreRow - Convert global row ID vector to Hypre format
96  // HypreSol - Convert solution vector to Hypre format
97  // HypreLhs - Convert left hand side matrix to Hypre format
98  // HypreSol - Convert right hand side vector to Hypre format
99  // FillSol - Fill/set solution vector
100  // FillLhs - Fill/set left hand side matrix
101  // FillRhs - Fill/set right hand side vector
102  // AsmSol - Assemble solution vector
103  // AsmLhs - Assemble left hand side matrix
104  // AsmRhs - Assemble right hand side vector
105  // Sol - Solve linear system
106  // Upd - Update solution vector (propagate new solution back to workers)
107  //
108  // Interpretation of the above DAG
109  // -------------------------------
110  // Control flow is from left to right and top to bottom.
111  //
112  // We start out with all worker chares contributing their porition of the
113  // global row IDs that each work on. This is labelled ChRow. The worker
114  // chares also contribute, in parallel, their portion of the global node
115  // ID lists at which they can set boundary conditions, labelled, ChBCs.
116  // Once both row IDs and BC node lists are received on LinSysMerger, the
117  // row IDs start being converted to a flat data structure understood by
118  // hypre, labelled HypreRow.
119  //
120  // The Ver partial synchronization point, besides HypreRow, also triggers
121  // two code paths verifying the received BCs as well as the row IDs. Both
122  // of these verification code paths only happen in DEBUG mode and skipped
123  // in RELEASE (optimized) mode. While VerRow is simply an Assert in
124  // rowsreceived(), VerBCs involves multiple steps of communication via
125  // multiple functions calls across three classes, LinSysMerger, host, and
126  // worker. While the VerBCs optional code path only happens in DEBUG mode,
127  // the code path does block on HypreLhs as well as on HypreRhs, which
128  // cannot proceed until VerBCs is finished. This is facilitated by
129  // 'trigger_ver_bc' which is only true by default, set in the constructor
130  // only in RELEASE mode; in DEBUG mode it must be triggered by
131  // host::doverifybc(), which is the last step of verifying the BCs.
132  //
133  // Once the row IDs are contributed, we then continue by all worker chares
134  // contributing their portion of the solution (unknown) vector, the
135  // nonzero values of the left hand side matrix, and the right hand side
136  // vector to those Linear System Merger group branch (one per CPU) they
137  // happened to reside on. These steps are denoted by ChSol, ChLhs, and
138  // ChRhs in the graph above, performed by the member functions charesol(),
139  // charelhs(), and charerhs(), respectively.
140  //
141  // Once each of the tasks, ChSol, ChLhs, ChRhs, (not all at the same time
142  // but separately) are done, we continue by converting these data
143  // structures to a format that Hypre expects, done in hypresol(),
144  // hyprelhs(), and hyprerhs(), denoted by similars labels in the graph.
145  // These functions basically flatten the C++ data structures to C-style
146  // arrays and linked arrays (for the matrix).
147  //
148  // Once the tasks, HypreSol, HypreLhs, HypreRhs, (not all at the same time
149  // but separately) are done and the hypre data structure for the row IDs
150  // are complete, we continue by assigning the flattened data structures
151  // (more precisely, their pointers) to the Hypre vectors and matrix. These
152  // are denoted FillSol, FillLhs, and FillRhs in the DAG and correspond to
153  // the functions sol(), lhs(), and rhs().
154  //
155  // Once the tasks, FillSol, FillLhs, FIllRhs, (not all at the same time,
156  // but separately) are, we continue by calling the Hypre "assembly"
157  // routines for the solution vector, the left hand side matrix, and the
158  // right hand side vector. These are denoted by AsmSol, AsmLhs, and AsmRhs
159  // in the DAG, and performed by the member functions assemblesol(),
160  // assemblelhs(), and assemblerhs(), respectively.
161  //
162  // Once the assembly of the solution (unknown) vector, the left hand side
163  // matrix, and the right hand side vector are all done (on a CPU), we call
164  // the solve() member function which solves the linear system calling
165  // Hypre and also updates the solution, i.e., propagates the new solution
166  // back to the worker chares.
167  //
168  // Note that Sol and Upd are both partial synchronization points, since
169  // they can happen at different times on different CPUs.
