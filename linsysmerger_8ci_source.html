70  // SDAG code follows. See http://charm.cs.illinois.edu/manuals/html/
71  // charm++/manual.html, Sec. "Structured Control Flow: Structured Dagger".
72 
73  // High-level overview of the dependency and asynchronous call structure
74  // ---------------------------------------------------------------------
75  //
76  // Directed Acyclic Graph (DAG):
77  // -----------------------------
78  //
79  // See src/LinSys/LinSysMerger.h.
80  //
81  // Interpretation of the DAG in src/LinSys/LinSysMerger.h
82  // ------------------------------------------------------
83  // Control flow is from left to right and top to bottom.
84  //
85  // We start out with all worker chares contributing their porition of the
86  // global row IDs that each work on. This is labelled ChRow. The worker
87  // chares also contribute, in parallel, their portion of the global node
88  // ID lists at which they can set boundary conditions, labelled, ChBCs.
89  // Once both row IDs and BC node lists are received on LinSysMerger, we do
90  // a global reduction to Transport::rowcomplete() spawns two broadcasts:
91  // (1) to LinSysMerger::rowsreceived(), and (2) to worker::setup().
92  //
93  // The member function tk::LinSysMerger::rowsreceived(), labeled Ver in
94  // the task-graph, triggers and performs a number tasks: (1) Verification
95  /// that ensures consistent global row IDs, labeled VerRow. This step only
96  // happens in DEBUG mode: VerRow is simply an Assert in member function
97  // rowsreceived). (2) Verification of the boundary contitions to be set
98  // that they match those set by the user, labeled VerBCs. While this step
99  // only happens in DEBUG mode, it involves multiple steps of communication
100  // via multiple functions calls across three classes, LinSysMerger, host,
101  // and worker. Note that this also involves multiple global reductions to
102  // host originating from both the worker chares as well as from the
103  // LinSysMerger group. Thus VerBCs is a global synchronization point and
104  // thus colored and filled on the task-graph. The VerBCs code-path
105  // culminates in tk::LinSysMerger::ver_complete(). Note both high and low
106  // order solution updates, labeled HighUpd and LowUpd, respectively,
107  // depend on ver_complete(). While this is is not strictly necessary from
108  // the correctness viewpoint, these dependencies are used to prevent the
109  // runtime system prematurely scheduling other reductions that these
110  // updates trigger in the worker chares, e.g., as a result of
111  // inciter::Carrier::aec() and inciter::Carrier::allowed(). As to why it
112  // is a problem to have multiple reductions in flight directed towards the
113  // same host objects see the discussion at
114  // https://lists.cs.illinois.edu/lists/arc/charm/2016-07/msg00033.html.
115  // (3) The third task tk::LinSysMerger::rowsreceived() initiates is
116  // labeled QueryBCVal. This initiates querying boundary condition values
117  // from worker chares. This step culminates is triggering the SDAG signals
118  // bcval_complete(), which is a prerequisite for setting boundary
119  // conditions on both left and right hand sides of both the low and the
120  // high order system. (4) The final step tk::LinSysMerger::rowsreceived()
121  // initiates is building Hypre data from the global row indices received,
122  // labeled HypreRow.
123 
124  // Once the row IDs are contributed, tk::Transporter::rowcomplete(), not
125  // only spaws tk::LinSysMerger::rowsreceived() but also issues a broadcast
126  // to the worker chares to start initializing their system. This is
127  // started with a call to the init() member function of the worker. The
128  // tasks include setting initial conditions, sending unknown/solution
129  // vectors for assembly to LinSysMerger, start computing the left and
130  // right hand sides for both the high and low order systems, and sending
131  // all this for linear system assembly. All worker chares contribute their
132  // portion of the solution (unknown) vector, the nonzero values of the
133  // left hand side matrix their portion of the diagonal lumped mass matrix,
134  // the right hand side vector for the high order system, as well as for
135  // the mass diffusion term that is added to the high order system to
136  // create the low order system in conjunction with the lumped mass matrix.
137  // The contribution goes to those Linear System Merger group branch (one
138  // per CPU) the workers happen to reside on. These steps are denoted by
139  // ChSol, ChLhs, and ChRhs in the graph above, and the
140  // contributions/assembly is performed by the member functions charesol(),
141  // charelhs(), and charerhs(), respectively.
142  //
143  // Once each of the tasks, ChSol, ChLhs, ChRhs, (not all at the same time
144  // but separately) are done, we continue by converting these data
145  // structures to a format that Hypre expects, done in hypresol(),
146  // hyprelhs(), and hyprerhs(), denoted by similars labels in the graph.
147  // These functions basically flatten the C++ data structures to C-style
148  // arrays and linked arrays (for the matrix). Note that the left and right
149  // hand sides can only begin to be converted to Hypre data format if the
150  // boundary conditions have also been applied on each. These are labeled
151  // LhsBC, RhsBC, and LumpBC and computed after QueryBCVal has finished.
152  //
153  // Once the tasks, HypreSol, HypreLhs, HypreRhs, (not all at the same time
154  // but separately) are done and the Hypre data structure for the row IDs
155  // are complete, we continue by assigning the flattened data structures
156  // (more precisely, their pointers) to the Hypre vectors and Hypre matrix.
157  // These are denoted FillSol, FillLhs, and FillRhs in the DAG and
158  // correspond to the member functions sol(), lhs(), and rhs().
159  //
160  // Once the tasks, FillSol, FillLhs, FIllRhs, (not all at the same time,
161  // but separately) are, we continue by calling the Hypre "assembly"
162  // routines for the solution vector, the left hand side matrix, and the
163  // right hand side vector. These are denoted by AsmSol, AsmLhs, and AsmRhs
164  // in the DAG, and performed by the member functions assemblesol(),
165  // assemblelhs(), and assemblerhs(), respectively.
166  //
167  // Once the assembly of the solution (unknown) vector, the left hand side
168  // matrix, and the right hand side vector are all done (on a CPU), labels
169  // AsmSol, AsmLhs, AsmRhs, respectively, we call the solve() member
170  // function which solves the high order linear system by calling Hypre and
171  // also updates the high order solution, i.e., propagates the new solution
172  // back to the worker chares, labeled HighUpd.
173  //
174  // The low order solution is performed once the high order right hand side
175  // vector, the mass diffusion term (to added to the right hand side), and
176  // the lumped mass matrix are all complete. This is a simple
177  // back-substitution, since the system is diagonal.
178  //
179  // Note that propagating the new hight and low order solutions only happen
180  // if not only the solutions are finished, but alse when the optional
181  // verification step is completed. As discussed above this is required so
182  // that we do not end up with multiple reductions in flight at the same.
183  // If these were not in place, the strict ordering of each CPU
184  // contributing to multiple reductions could not be guaranteed. See also
185  // the discussion on the charm email list linked above.
186  //
187  // A word on multiple invokation of SDAG triggers: As the commit message
188  // of b79ea29 explains, the "triggers", such as rhs_complete(), enumerated
189  // below, at threir call site should be thought of as messages being put
190  // in some message queue and consumed at when clauses wherever they
191  // appear. If they appear in multiple when clauses, they consume multple
192  // message, thus they have to be triggered multiple times. See also the
193  // commit message of b79ea29 and
194  // https://lists.cs.illinois.edu/lists/arc/charm/2016-09/msg00002.html.
