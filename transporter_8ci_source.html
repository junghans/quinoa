40  // SDAG code follows. See http://charm.cs.illinois.edu/manuals/html/
41  // charm++/manual.html, Sec. "Structured Control Flow: Structured Dagger".
42 
43  // High-level overview of the dependency and asynchronous call structure
44  // ---------------------------------------------------------------------
45  //
46  // Directed Acyclic Graph (DAG):
47  // -----------------------------
48  // DAG legend:
49  // Row -- Init Row - linear system rows complete
50  // Msum - mesh surrounding mesh
51  // Msum -- Setup Init - initialization continues
52  // Setup - initialize mesh IDs, etc.
53  // Diag - compute diagnostics
54  // Out - Output particles to file
55  // ParCom -- Npar - Out -- Eval ParCom - particle communications
56  // / Eval - evalaute time step
57  // Diag Npar - output number of particles
58  //
59  //
60  // Interpretation of the above DAG
61  // -------------------------------
62  // Control flow is from left to right and top to bottom.
63  //
64  // First graph: If the linear system merger branches have done their part
65  // of storing and exporting global row ids (Row), initialization can
66  // continue. If the mesh chunks surrounding mesh chunks data structure has
67  // been built (Msum), the setup can continue.
68  //
69  // Second graph (happening after the first graph): At the final time step
70  // stage particles are advanced on each Carrier chare followed by
71  // searching for the mesh cells of each particle, which may require
72  // communication if some particles have left the mesh cunk held by
73  // Carrier chares. After the particle communication has been completed
74  // (ParCom), each Carrier chare sends its number particles (Npar) the it
75  // will contribute to the particle output in the given time step. Once
76  // that is done particles are sent and buffered for output with done by
77  // ParticleWriter. In the meantime diagnostics are also computed by the
78  // linear system merger group objects on each PE. The time step is
79  // evaluated only if both the diagnostics and the particle output have
80  // been done.
81  //
82  // Collecting diagnostics is optional, i.e., it does not happen every time
83  // step. If diagnostics are collected diag_complete() is signaled from the
84  // reduction target, Transporter::diagnostics(). If diagnostics are not
85  // collected in a time step, collection is skipped, and diag_complete() is
86  // signaled from the shortcut function, Transporter::diagcomplete().
87  //
88  // Similar to diagnostics output, particle output is also optional and
89  // does not happen every time step. The Charm++ reduction target
90  // Transporter::outcomplete() is reached either way but on different
91  // routes: if particles are output, the reduction originates from the
92  // ParticleWriter Charm++ group after writeCoords() finished the output,
93  // if particles are not output, the reduction originates from
94  // Carrier::out(). Either way, the above DAG and its logic remains the
95  // same.
